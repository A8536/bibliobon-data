# Справочник полей для библиографического парсера

Черновик для редакторской правки.

Документ описывает, как парсер должен раскладывать входное библиографическое
описание по полям редакторской базы Bibliobon. Это не машинный контракт и не
инструкция по публичному выводу. После редакторской правки на его основе можно
подготовить строгий англоязычный машиночитаемый вариант.

Основные ориентиры:

- ГОСТ Р 7.0.100-2018: библиографическое описание ресурса, его части и
  идентифицирующего ресурса;
- ГОСТ Р 7.0.80-2023: заголовок библиографической записи и точки доступа;
- предметная область Bibliobon: книги, статьи, журналы, газеты, сборники,
  многотомники, материалы конференций, законодательные и нормативные акты,
  электронные ресурсы.

Не включаем в активную модель специальные поля для нотных, картографических,
патентных, аудио- и видеоизданий, если они не нужны конкретной записи. Такие
сведения временно можно сохранять в `notes` или `publication_details`.

## Общий принцип для парсера

Входная строка может описывать:

1. самостоятельный источник;
2. статью или часть ресурса;
3. контейнер статьи: журнал, номер газеты, выпуск журнала, сборник,
   материалы конференции, том многотомника;
4. одновременно статью и контейнер.

Если строка содержит разделитель вида `//`, сначала разбирается часть до `//`
как статья (`Work` + `Article`), затем часть после `//` как идентифицирующий
ресурс: журнал/газета (`Journal` + `JournalIssue`) или сборник/том
(`Work` с `work_type=container` + `Article.container_work`).

Данные статьи не должны дублировать данные контейнера. Город, издательство,
год выпуска, номер выпуска, название журнала или сборника должны быть у
контейнера. У статьи остаются собственные авторы, заглавие, сведения об
ответственности, DOI/URL, примечания к статье и страницы внутри контейнера.

Если статья напечатана в нескольких выпусках, это не должно сжиматься в одно
поле номера выпуска. Правильная будущая модель: одна статья и несколько
размещений (`ArticlePlacement`) в разных выпусках. Пока это место помечается
для ручной проверки.

## Новые поля, которые стоит добавить в модель

Эти поля пока могут отсутствовать в базе. Они нужны парсеру и редактору, чтобы
не смешивать разные части ГОСТ-описания.

| Поле | Где нужно | Назначение |
| --- | --- | --- |
| `multipart_statement` | `Work`, `Source`, возможно `Issue` | Общая фраза о многочастности: `в 2 т.`, `в 3 ч.`, `в 10 вып.`. Это не номер конкретного тома, а характеристика всего ресурса. |
| `publication_status_note` | `Work`, `Source` | Фразы о статусе публикации: `на правах рукописи`, `самиздат`, `рукопись`, `неопубликованный доклад`. |
| `translation_note` | `Work`, `Source` | Сведения о переводе без конкретного имени переводчика: `пер. с нем.`, `пер. с англ.`. Если указан переводчик, запись должна идти в `responsibility_statement` и связь с автором/лицом. |
| `event_name` | `Work`, `Source`, `Issue` | Название конференции, чтений, заседания или события, если оно нужно отдельно от заглавия. |
| `event_date` | `Work`, `Source`, `Issue` | Дата события или доклада: `докл. 19.04.95`, `13-15 октября 2021 г.`. |
| `event_place` | `Work`, `Source`, `Issue` | Место события: `Санкт-Петербург`, `Москва`. |
| `legal_document_number` | `Work`, `Source` | Номер законодательного или нормативного акта: `Федеральный закон № 69-ФЗ`, `ГОСТ Р 7.0.100-2018`. |
| `legal_status_statement` | `Work`, `Source` | Сведения о принятии, утверждении, введении в действие: `принят Государственной думой...`, `утвержден...`, `дата введения...`. |
| `access_mode` | `Work`, `Source` | Режим доступа к электронному ресурсу, если не свободный: `для зарегистрированных пользователей`, `по подписке`. |
| `source_of_title_note` | `Work`, `Source` | Источник заглавия электронного или нетитульного ресурса: `загл. с титул. экрана`, `загл. с обложки`. |

До добавления этих полей парсер должен сохранять такие фрагменты в `notes` или
`publication_details` и ставить `description_status=needs_review`.

## Таблица `Works`

**Название:** библиографические записи.

**Назначение:** основная редакторская таблица источников. В ней хранятся
самостоятельные книги, статьи, сборники, конкретные тома, выпуски, электронные
публикации и другие описываемые единицы. Для парсера это главная таблица,
куда попадает описание самого цитируемого объекта.

### Идентификация и тип

| Поле | Название | Назначение и правила |
| --- | --- | --- |
| `work_id` | стабильный ID записи | Внутренний стабильный идентификатор. Парсер не должен менять существующие ID. Для новых записей ID создаётся системой. |
| `source_django_id` | старый ID сайта | Поле совместимости со старой базой. Парсер его не заполняет для новых записей. |
| `source_sequence` | порядок исходной записи | Технический порядок старого импорта. Для новых записей обычно пусто. |
| `source_number` | номер источника | Исторический номер печатного указателя или синтетический номер для новых записей. Парсер не должен имитировать исторические номера. |
| `work_type` | тип записи | `book` для самостоятельного источника, `article` для статьи, `container` для сборника/тома/контейнера, `unknown` если парсер не уверен. |
| `description_status` | статус разбора | `parsed` если запись уверенно разложена, `partial` если часть полей разобрана, `raw_only` если почти всё оставлено сырой строкой, `needs_review` если нужна ручная проверка. |

### Заглавие

| Поле              | Название                         | Назначение и правила                                                                                                                                                                                                                                                                                                         |
| ----------------- | -------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `title`           | основное заглавие                | Главное название описываемого объекта. Для статьи это название статьи, не журнала и не сборника. Не включать авторов, редакторов, город, издательство, год, страницы, серию, номер тома или выпуска. Примеры: `Бумажные денежные знаки России`; `Основы финансовой системы СССР`.                                            |
| `parallel_title`  | параллельное заглавие            | Эквивалент основного заглавия на другом языке или в другой графике. Если в строке есть `= Russian Paper Money`, сохранять без знака `=`. Не использовать для вариантов русского названия.                                                                                                                                    |
| `subtitle`        | сведения, относящиеся к заглавию | Краткое пояснение к названию: подзаголовок, тема. Использовать для текста после основного заглавия, который раскрывает содержание или тип издания. Пример: `Государственные выпуски с 1769 г.` или `сборник материалов конференции, 13-15 октября 2021 г., Санкт-Петербург`.<br>                                             |
| `title_remainder` | прочие сведения к заглавию       | Длинные или сложные сведения из области заглавия: описание конференции, подзаголовок с датами и местом события, уточнение сайта. <br>Сюда могут попадать: `каталог`, `учебное пособие`, `справочник`, `тезисы докладов`, если это часть заглавия/подзаголовка.<br>Не помещать сюда редакторов, серию и издательские данные.  |
| `volume_number`   | номер тома                       | Номер конкретного тома описываемой записи: `Т. 1`, `Т. 2`, `Vol. 3`. Если запись говорит обо всём издании `в 2 т.`, это не `volume_number`, а предлагаемое поле `multipart_statement`.                                                                                                                                       |
| `part_number`     | номер части                      | Номер конкретной части: `Ч. 1`, `Ч. 2`. Не смешивать с номером выпуска журнала.                                                                                                                                                                                                                                              |
| `part_title`      | заглавие части или тома          | Частное заглавие конкретного тома, части или выпуска. Пример: `Античность` в записи `[Вып.] 1: Античность`.                                                                                                                                                                                                                  |
| `host_title`      | сырое заглавие контейнера        | Legacy/raw поле. Для новых разборов не использовать как нормальную связь. Название журнала, газеты, сборника или тома должно переноситься в `Journal`, `JournalIssue` или контейнерный `Work`.                                                                                                                               |

### Ответственность и авторы

| Поле                       | Название                          | Назначение и правила                                                                                                                                                                                                                   |
| -------------------------- | --------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `raw_author_string`        | исходная строка авторов           | Сырой авторский фрагмент из исходного описания. Использовать для контроля, когда авторы ещё не нормализованы. Не считать основным нормальным способом хранения авторов.                                                                |
| `responsibility_statement` | сведения об ответственности       | Текст после `/` и последующие группы после `;`: авторы в форме источника, редакторы, составители, переводчики, организации. Примеры: `составитель И. П. Иванов`; `перевод с немецкого А. А. Петрова`; `под редакцией А. А. Молчанова`. |
| `responsibility_note`      | legacy-примечание ответственности | Старое поле. Для новых разборов не использовать, предпочтительно `responsibility_statement`.                                                                                                                                           |

Нормализованные авторы и лица создаются через `Authors` и `WorkAuthors`.
`responsibility_statement` сохраняет печатную форму, а `WorkAuthors` даёт
упорядоченные связи для поиска и контроля.

### Издание

| Поле | Название | Назначение и правила |
| --- | --- | --- |
| `edition_statement` | сведения об издании | Сведения вроде `2-е изд., испр. и доп.`, `Изд. 4-е`, `7-е изд., стер.`. Не помещать сюда редакторов, если это не ответственность за издание. |
| `additional_edition_statement` | дополнительные сведения об издании | Дополнительное уточнение после основных сведений об издании, если оно отделено в источнике. Использовать редко. |

### Публикация, производство, распространение

| Поле                | Название            | Назначение и правила                                                                                                                                                   |
| ------------------- | ------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `publication_place` | место публикации    | Город или места публикации: `Москва`, `Санкт-Петербург ; Москва`, `[Б. м.]`, `[Санкт-Петербург?]`. У статьи обычно пусто, если место относится к журналу или сборнику. |
| `publisher`         | издатель            | Издательство или издающая организация: `Наука`, `Проспект`, `[б. и.]`. У статьи обычно пусто, если издатель относится к контейнеру.                                    |
| `publication_date`  | дата публикации     | Текстовая дата публикации: `2017`, `2017, 22 дек.`, `1997-1999`, `сор. 2003`. Для года дополнительно заполняется `inferred_year`.                                      |
| `inferred_year`     | нормализованный год | Числовой год для поиска и сортировки. Если в строке диапазон, обычно брать первый год и оставить полную форму в `publication_date`.                                    |
| `manufacture_place` | место изготовления  | Типография или место изготовления, если отдельно указано и важно.                                                                                                      |
| `manufacturer`      | изготовитель        | Типография/изготовитель, если отличается от издателя.                                                                                                                  |
| `manufacture_date`  | дата изготовления   | Дата изготовления, если отдельно указана.                                                                                                                              |
| `copyright_date`    | дата copyright      | Дата copyright, если приведена отдельно: `cop. 2018`.                                                                                                                  |

### Физическая характеристика

| Поле | Название | Назначение и правила |
| --- | --- | --- |
| `extent` | объём | Количество страниц, листов, томов как физический объём конкретного источника: `158 с.`, `[14], 264 с.`, `1 электрон. опт. диск`. Не помещать сюда тираж, иллюстрации, размер и примечания. |
| `illustrations` | иллюстрации | Наличие иллюстраций, таблиц, портретов, диаграмм: `ил.`, `табл.`, `портр.`, `диагр.`. |
| `dimensions` | размер | Физический размер: `20 см`, `120x250 мм`. |
| `accompanying_material` | сопроводительный материал | Материал после `+`: `+ 1 электрон. опт. диск`, `+ прил. (7 с.)`. |
| `circulation` | тираж | Тираж: `3000 экз.`, `100 экз.`. Не хранить в `extent`. |
| `physical_description` | сырое физическое описание | Legacy/raw поле для неразобранной физической характеристики. Парсер должен по возможности раскладывать в `extent`, `illustrations`, `dimensions`, `accompanying_material`, `circulation`. |

### Страницы статьи

| Поле | Название | Назначение и правила |
| --- | --- | --- |
| `article_pages` | страницы статьи | Сырой диапазон страниц статьи внутри контейнера: `С. 17-28`, `С. [1-8]`, `С. I-XXXVI, 1-12`. Для целевой модели предпочтительно хранить в размещении статьи (`Article.pages_raw` / `ArticlePlacement.pages_raw`), но поле оставлено для удобства редактора. |
| `page_start` | первая страница | Нормализованная первая арабская страница, если её можно извлечь без потери смысла. |
| `page_end` | последняя страница | Нормализованная последняя арабская страница, если её можно извлечь без потери смысла. |

### Серия, многочастность, примечания

| Поле                  | Название                | Назначение и правила                                                                                                                                                                  |
| --------------------- | ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `series_statement`    | серия                   | Серия или подсерия в круглых скобках описания: `Библиотека газеты «Всемирный коллекционер»`, `Актуальное законодательство`. Не использовать для сборника, в котором находится статья. |
| `notes`               | общие примечания        | Общие примечания: `на правах рукописи`, `самиздат`, `загл. с титул. экрана`, `докл. 19.04.95`, если нет отдельного поля. Также временное место для сомнительных фрагментов.           |
| `bibliography_note`   | библиография            | Legacy поле примечание о библиографии: `Библиогр.: с. 125-132`, `Библиогр. в примеч.`.                                                                                                |
| `index_note`          | указатели               | Legacy поле примечание об указателях: `Имен. указ.: с. 133-135`.                                                                                                                      |
| `contents_note`       | содержание              | Legacy поле содержание, если нужно сохранить состав сборника или группы материалов.                                                                                                   |
| `publication_details` | сырая строка публикации | Legacy/raw поле для неразобранной части описания. Парсер должен использовать его только как временное хранилище при неуверенности.                                                    |
| `public_review`       | публичная заметка       | Текст описания или редакторская аннотация, не часть ГОСТ-описания. Заполнять, если указана в источнике данных.                                                                        |

### Идентификаторы и электронные ресурсы

| Поле | Название | Назначение и правила |
| --- | --- | --- |
| `isbn` | ISBN | ISBN книги, сборника или выпуска. |
| `issn` | ISSN | ISSN сериального ресурса или выпуска, если указан. Обычно лучше хранить у `Journal`, если относится к журналу в целом. |
| `doi` | DOI | DOI статьи или электронного ресурса. |
| `url` | URL | Электронный адрес самой записи или статьи. Для статьи в электронном журнале это URL статьи, не главной страницы журнала. |
| `access_date` | дата обращения | Дата обращения к сетевому ресурсу. |
| `content_type` | вид содержания | Например `Текст`, `Изображение`, `Текст. Изображение`. Для нашей базы чаще всего `Текст`. |
| `media_type` | характеристика доступа | Например `электронный`, `непосредственный`. |
| `carrier_type` | носитель | Использовать при необходимости: `online`, `CD-ROM`, `печатный`. |

## Таблица `Authors`

**Название:** авторы и ответственные лица/организации.

**Назначение:** нормализованные точки доступа: авторы, редакторы,
составители, переводчики, организации, временные организации и конференции,
если они используются как заголовок или точка доступа.

| Поле             | Название                | Назначение и правила                                                                                                                 |
| ---------------- | ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| `author_id`      | ID автора               | Стабильный ID. Парсер не меняет существующий ID.                                                                                     |
| `display_name`   | отображаемое имя        | Читаемая форма: `Иванов И. И.`, `Российская Федерация. Законы`, `Нумизматические чтения`.                                            |
| `heading_name`   | заголовочная форма      | Форма заголовка по ГОСТ Р 7.0.80-2023, если отличается от display_name. Для законов и организаций может быть важнее, чем личное имя. |
| `sort_name`      | форма сортировки        | Нормализованная строка сортировки.                                                                                                   |
| `aliases`        | варианты имени          | Другие написания, псевдонимы, сокращения.                                                                                            |
| `person_dates`   | даты лица               | Legacy поле. Годы жизни или существования, если нужны для идентификации.                                                             |
| `authority_note` | авторитетное примечание | Legacy поле. Источник нормализации, уточнения, неоднозначность.                                                                      |
| `note`           | примечание              | Редакторская заметка.                                                                                                                |

## Таблица `WorkAuthors`

**Название:** связь записи с авторами и ответственными лицами.

**Назначение:** хранит порядок и роль лиц/организаций, связанных с записью.
Не заменяет `responsibility_statement`, потому что ГОСТ-описание должно
сохранять печатную форму ответственности.

| Поле | Название | Назначение и правила |
| --- | --- | --- |
| `work_id` | запись | Ссылка на `Works`. |
| `author_id` | автор/лицо/организация | Ссылка на `Authors`. |
| `sort_order` | порядок | Порядок в описании или порядок точки доступа. |
| `role` | роль | `author`, `editor`, `compiler`, `translator`, `organization`, `conference`, `subject_person` и т.п. |
| `source_text` | исходный текст | Фрагмент исходной строки, из которого создана связь. |
| `name_as_printed` | имя как напечатано | Форма имени в источнике, если отличается от нормализованной. |
| `include_in_responsibility` | включать в ответственность | Истина, если лицо должно участвовать в формировании `responsibility_statement`. |
| `is_primary_heading` | основной заголовок | Истина, если это основной заголовок библиографической записи. |

## Таблица `Journals`

**Название:** журналы, газеты и продолжающиеся издания.

**Назначение:** хранит идентичность сериального ресурса в целом. Сам журнал не
является цитируемой библиографической записью; цитируются статьи или конкретные
выпуски.

| Поле                       | Название                  | Назначение и правила                                                                      |
| -------------------------- | ------------------------- | ----------------------------------------------------------------------------------------- |

## Сборники и тома как контейнеры

Сборник, материалы конференции, ежегодник, том многотомника и конкретный
выпуск продолжающегося издания без периодичности хранятся как `Works` с
`work_type=container`.

Правила:

- если статья входит в сборник, создаётся отдельная запись статьи (`Work`) и
  отдельная запись контейнера (`Work` с `work_type=container`);
- связь статьи со сборником идёт через `Article.container_work` или
  `ArticlePlacements.container_id`;
- город, издательство, год, серия, ISBN, физическое описание сборника хранятся
  у контейнера, а не у статьи;
- если в базе есть только одна статья из сборника, полная структура всё равно
  допустима, но интерфейс может показывать её как обычную ссылку без лишних
  инструментов.

## Как парсер должен отделять статью от контейнера

### Статья в журнале

Примерная схема:

```text
Автор статьи. Название статьи / сведения об ответственности статьи. — Текст : электронный // Название журнала. — 2018. — № 1. — С. 10-15. — URL: ...
```

Разложение:

- статья: `Works.title`, `WorkAuthors`, `Works.responsibility_statement`,
  `Works.doi`, `Works.url`, `Works.access_date`;
- журнал: `Journals.title`, `Journals.title_remainder`, `Journals.issn`;
- выпуск: `JournalIssues.year`, `publication_date`, `issue_number`, `volume`,
  `gross_number`, `title`;
- размещение: `Article.pages_raw`, `page_start`, `page_end`;
- не переносить в статью город и издательство журнала, если они относятся к
  журналу или выпуску.

### Статья в газете

Газета хранится как `Journal`, конкретный номер как `JournalIssue`.

Особенно важны:

- `JournalIssues.publication_date`: `22 дек. 2017`, `20-26 сент. 2017`;
- `JournalIssues.issue_number`: номер газеты, если есть;
- `Article.pages_raw`: страницы.

### Статья в сборнике

Примерная схема:

```text
Автор статьи. Название статьи // Название сборника : подзаголовок / под ред. ... — Москва : Издательство, 2021. — С. 20-31.
```

Разложение:

- статья: отдельный `Work` с `work_type=article`;
- сборник: отдельный `Work` с `work_type=container`;
- связь: `Article.container_work` / `ArticlePlacements.container_id`;
- страницы: в размещении статьи;
- город, издательство, год, серия, ISBN, объём сборника: у контейнера.

### Доклад на конференции

Если доклад опубликован в сборнике, это статья в сборнике. Если есть только
сведения об устном докладе без публикации, это самостоятельный `Work` с
примечанием или предлагаемыми полями `event_name`, `event_date`, `event_place`.

Фрагмент `докл. 19.04.95` до добавления специальных полей сохранять в `notes`
и ставить `needs_review`, если неясно, является ли это датой события или
примечанием к публикации.

### Законодательные и нормативные акты

Если запись описывает отдельное издание закона или стандарта, это
самостоятельный `Work`.

Рекомендуемое разложение:

- `title`: собственное название акта;
- `subtitle` или предлагаемое `legal_document_number`: вид и номер:
  `Федеральный закон № 69-ФЗ`, `ГОСТ Р 7.0.100-2018`;
- `title_remainder` или предлагаемое `legal_status_statement`: принятие,
  утверждение, дата введения;
- публикация, объём, серия, тираж, ISBN: обычные поля `Work`.

### Электронный ресурс

Для сетевого ресурса:

- `url` обязателен, если это источник из сети;
- `access_date` обязателен для сетевых ресурсов, если нет устойчивой даты
  публикации или если нужна проверяемость ссылки;
- `content_type`, `media_type`, `carrier_type`: использовать для различения
  `Текст : электронный`, `Текст : непосредственный`;
- `source_of_title_note`: желательно добавить для `загл. с титул. экрана`,
  пока хранить в `notes`;
- `access_mode`: желательно добавить для `по подписке`, `для зарегистрированных
  пользователей`, пока хранить в `notes`.
